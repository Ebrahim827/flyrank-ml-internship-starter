Content Decay & Refresh Scoring:
An Honest Baseline-to-Model Audit

Does a learned model actually beat a simple, readable rule at flagging which pages deserve a human's attention? We built both, tested them the honest way, and report what actually happened.

01 Title & Abstract

Does the model beat the rule?

This project asks a simple operational question: out of a large content portfolio, which pages are worth a person's time to review for refresh, and can a learned model meaningfully improve on a transparent, hand-written rule? We built a baseline rule from two audited signals — content staleness and traffic volume — then trained Logistic Regression and Random Forest classifiers on the same task, evaluating everything with a client-grouped train/test split so the model could not simply memorize individual clients. Under that honest split, the rule outperformed both learned models at every threshold we tested, and a parallel leakage test showed that a naive random split had been inflating apparent model skill from a genuine 0.55 up to a hollow, memorized 1.00. We report these results directly rather than reframing them, translate the rule into a three-category action playbook covering roughly 15% of the portfolio, and treat every claim here as observational and decision-support only — not causal, and not a guarantee about any individual page.

02 Introduction

The decision this supports

Content and SEO teams typically manage far more pages than they can manually review on any regular cadence. The practical question isn't "what will happen to this page" — it's narrower and more useful: given limited review time, which pages should a person look at first?

This work treats that as a triage problem. Before reaching for a machine-learned model, we asked whether a simple, fully readable rule could already do the job — and then held any learned model to the standard of actually beating that rule, on the same data, under the same honest test.

03 Data

What this is built on

The analysis uses the FlyRank ML Internship starter dataset: 30,000 anonymized content items across 32 pseudonymized clients, each row a trailing-90-day snapshot of search and engagement metrics (impressions, clicks, sessions, click-through rate, average search position, and related fields).

What we excluded, and why

  • Trend/decline fields (trend_direction, trend_pct) were excluded from every model and rule input. The label we predict is derived directly from these fields, so using them as inputs would mean the model was shown the answer — this is confirmed directly in the leakage audit below.
  • Client and content identifiers are pseudonymous codes, used only to define a grouped train/test split — never fed to any model as a feature.
  • Rows with avg_position = 0 are treated as "no ranking data available," not as an actual top rank of zero, per the dataset's documented convention.
04 Methodology

Rule, model, and how we tested both honestly

The label

A page is treated as "declining" if its documented trend direction is downward in this trailing-90-day snapshot. This is an observed pattern in this dataset and time window — not a forecast of future decline.

The baseline rule

A page is flagged for refresh review if, and only if, it meets both conditions: it has gone 180+ days without an update, and it is still receiving meaningful traffic (500+ impressions in the trailing 90 days). Both signals were checked against the data before being used — staleness and traffic volume each showed a real but non-monotonic relationship with decline (bucketed comfortably above the base rate in the middle ranges, but not a clean, one-directional trend end to end), which is why the rule requires both together rather than relying on either alone.

The learned models

Logistic Regression and Random Forest classifiers were trained on the same feature set (traffic, engagement, position, and static content attributes — the same fields available to the rule, plus a few more), predicting the same declining/not-declining label.

Validation design

All model comparisons use a client-grouped split: every page belonging to one client is entirely in the training set or entirely in the test set, never both. A plain random row-level split would let a model partially learn a specific client's quirks and then appear to "recognize" that same client in testing — a form of memorization rather than genuine pattern-learning.

Leakage audit

Every feature was checked against two questions: is it knowable before the moment of prediction, and is it free of any mathematical link to the label? Trend-based fields failed this check and were excluded. As a direct test of the audit itself, we deliberately re-added the excluded field (trend_pct) and retrained — precision jumped from 0.55 to a perfect 1.00, confirming both that the field really would have been leakage, and that our leakage test is capable of catching it.

What the model leaned on

Permutation importance on the Random Forest surfaced days_with_impressions as the strongest single feature (shuffling it cost ~3.4 accuracy points), followed by content_age_days and avg_position, with a cluster of traffic/engagement features trailing behind. No single feature dominated overwhelmingly — a healthier sign than the near-total dominance that would suggest hidden leakage.

06 Limitations & honest framing

What this evidence does not support

  • This is cross-sectional data — one snapshot, no intervention was run. Nothing here supports "refreshing a page will increase its traffic." The honest framing is decision-support: these pages look worth reviewing first, based on observed patterns.
  • No claim is made about Google's ranking algorithm. Nothing here was modeled against, or claims to explain, how any search engine ranks pages.
  • The 180-day / 500-impression thresholds in the rule are reasoned from the signal checks, not formally optimized — a different dataset or business context could reasonably use different cutoffs.
  • The staleness and volume signals were each found to be directionally associated with decline, but non-monotonic — neither is a clean, standalone predictor, which is exactly why the rule requires both together.
  • This is a snapshot analysis on 30,000 rows across 32 clients — it is not a continuously monitored system, and results should be re-run periodically rather than trusted indefinitely.
07 Ranked recommendations

The action playbook

Because the honest evaluation favored the simple rule, the playbook below is built on the rule, not the learned model. Applied to the full 30,000-row dataset, 4,436 pages (about 15% of the portfolio) received one of three action flags; the remainder received no action.

stale_but_visible

Refresh review

180+ days since update, still receiving 500+ impressions in the last 90 days. Worth a human read to confirm the page still matches current offerings before a rewrite.

ctr_underperforming_for_position

CTR-fix review

Ranks well, but click-through rate sits well below what similar pages at that position typically get. Worth a look at the title and search snippet.

strong_performer

Protect & monitor

Fresh, well-ranked, high-traffic. No action needed now — just periodic re-checking, since strong performance is not permanent.

Human review is required — this is not an automation pipeline

Every flag above is a starting point for a person, not an instruction to a system. Before acting: read the actual page, confirm the flag still makes sense in context, and check for an obvious external explanation (seasonality, a since-launched competitor, a business change) before spending review time.

What should never be automated: publishing a rewritten page without human review; deleting a page because it's unflagged; treating any flag as a guarantee of a traffic outcome; acting on flags for legal, medical, financial, or safety-sensitive content without domain-expert review; trusting a CTR-fix flag on very low-traffic pages, where small sample counts can look extreme by chance alone.

Monitoring / retrain triggers

Re-run this analysis on a regular cadence (monthly is a reasonable default) rather than treating any single run as permanent. Re-check sooner if: the overall base rate of decline shifts noticeably from ~0.56; a large batch of pages gets refreshed at once (which changes "days since update" broadly); or the count of flagged pages changes sharply without a clear cause — any of these suggest the underlying data has shifted enough that the current thresholds may no longer fit.
